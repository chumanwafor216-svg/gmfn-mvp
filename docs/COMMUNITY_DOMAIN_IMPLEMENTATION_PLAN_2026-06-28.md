## 1. Product Split

GSN now has two community concepts.

### Type 1 - Community

A `Community` is the lightweight social/invite relationship layer.

It supports:

- create a community;
- invite friends or known members;
- join an existing community;
- basic membership and participation;
- ordinary social trust context.

This path is free or lightweight. It does not require a purchased domain,
formal governance, department hierarchy, renewal, or institutional operating
space.

### Type 2 - Community Domain

A `Community Domain` is the paid institutional operating space for real-world
organized bodies.

It supports:

- legal or public organizational identity;
- domain name and domain ownership inside GSN;
- owner/admin governance;
- branches, departments, classes, committees, zones, or sections;
- recurring subscription or package activation;
- module settings for GSN services;
- member approval and role assignment;
- evidence, verification, analytics, and trust propagation.

The user-facing entrances should stay separate:

```text
Create Community
Purchase Community Domain
```

Optional third entrance for later:

```text
Upgrade Existing Community to Community Domain
```

That third entrance matters because some existing lightweight communities may
later become formal churches, associations, schools, unions, cooperatives, or
market bodies inside GSN.

## 2. Naming Contract

Use these names consistently.

| Name | Meaning |
| --- | --- |
| Community | Free/lightweight social unit. |
| Community Domain | Durable institutional object bought/owned by an organization. |
| Community Domain Package | The commercial subscription/package that activates or renews the domain. |
| Verified Community Domain | A stronger verified/protected state of a Community Domain. |
| Community Domain Engine | The operating engine that starts after a Community Domain exists. |
| Community Package | Existing bundle language; should be phased away or scoped to specific feature bundles, not used as the object name. |

Devil's advocate:

- Calling the product `Community Package` will confuse users and coders,
  because a package sounds like a payment bundle, not an operating home.
- Calling everything simply `Community` will hide the most valuable SaaS line:
  schools, churches, unions, branches, markets, cooperatives, clubs, and
  associations paying for a governed digital home.
- Hard-coding separate engines for school, church, union, and market would
  make the system expensive and brittle. The correct object is one configurable
  Community Domain with generic hierarchy nodes and module settings.

## 2.1 Research Principles Applied to GSN

The research does not tell GSN to become an academic system. It gives practical
rails for building a serious institutional product.

### Boundaries

Ostrom's work on self-governing institutions points to the importance of clear
membership and resource boundaries. For GSN this means a Community Domain must
know:

- who owns the domain;
- who belongs;
- what branch, department, class, committee, or node a member belongs to;
- which services are active;
- which evidence can be shown publicly;
- what stays private.

Implementation implication:

- add durable domain status, membership status, node membership, and
  visibility-policy fields instead of relying only on the existing community
  name and membership row.

### Local Fit

Ostrom also warns against one-size-fits-all governance. The same lesson appears
in online-community governance research: communities have different values and
strictness levels, so generic moderator/admin defaults are not enough.

Implementation implication:

- use one engine, but give each domain a `template_key`, `domain_type`,
  module settings, role presets, and policy presets.

### Real Work Beats Org Charts

Brown and Duguid's communities-of-practice research shows that how people
actually work can differ from formal job descriptions and org charts.

Implementation implication:

- do not model only formal structure. Also record practice/evidence: meetings,
  contributions, witness events, verification, service usage, trust relay, and
  member activity.

### Community Is Relationship, Not Only Place

Preece and Maloney-Krichmar describe online community as broader than physical
proximity; relationships, activity, and shared purpose matter.

Implementation implication:

- a Community Domain should support churches with branches, diaspora unions,
  professional bodies, alumni groups, and schools whose members are not all in
  one place.

### Policy Is Bigger Than Roles

PolicyKit argues that most platforms encode a narrow admin/moderator permission
model, while real communities need procedures for decisions.

Implementation implication:

- keep simple roles for MVP, but add a policy/action-review spine so later
  decisions can require owner approval, admin approval, node-admin approval,
  member witness, vote/quorum, or appeal.

### Appeals Matter

Community-led moderation research points to structured appeals as a way to
balance fairness, accountability, and local values.

Implementation implication:

- add a basic appeal/dispute model early, even if the first UI is small:
  rejected membership, removed member, blocked post, revoked evidence, or
  disputed role assignment should have a traceable review path.

### Evidence Must Be Verifiable but Not Overclaimed

W3C Verifiable Credentials separates issuer, holder, verifier, evidence,
validity, and reliance policy. It also states that verifiability does not by
itself prove the truth of every claim.

Implementation implication:

- GSN Community Domain evidence should carry issuer, subject, validity,
  evidence source, status/currentness, and reader boundary language.
- GSN must continue saying that evidence supports judgement; it is not legal
  certification, loan approval, payment instruction, or authority to release
  goods or money.

## 3. Community Domain Package Lifecycle

The institutional package should follow this lifecycle:

1. Draft
   - User starts `Purchase Community Domain`.
   - GSN creates a draft intent, not a live domain.
   - The draft can collect name, type, owner, country/state, and intended use.

2. Domain Check
   - User chooses a domain name or display identity.
   - Backend checks uniqueness, reserved words, impersonation risk, and
     blocked terms.

3. Package Quote
   - Backend returns the package offer, included modules, price, renewal
     period, and limits.
   - This should be a Community Domain Package, not a generic community package.

4. Payment Instruction
   - GSN creates a payment instruction or expected payment.
   - Payment instruction does not activate the domain by itself.
   - Activation requires confirmed payment or an approved admin override.

5. Provisioning
   - Backend creates or upgrades the underlying community record.
   - Backend creates the Community Domain record.
   - Backend creates default service settings, owner role, admin role, member
     role, and default root node.

6. Configuration
   - Owner defines structure: departments, branches, classes, zones,
     committees, sections, chapters, or similar.
   - Owner chooses membership rules and enabled modules.

7. Launch
   - Domain becomes active.
   - Members can request membership or enter by invite.
   - Admins can approve, assign roles, and manage nodes.

8. Renewal / Suspension
   - Domain has renewal date and subscription state.
   - Expired domains should keep evidence history but restrict paid operating
     tools until renewed.

## 4.1 Connect-the-Dots Map to Existing Code

The right approach is not a rewrite. It is adding the missing institutional
layer around the useful structures already present.

| Existing spine | Keep using it for | Add beside it |
| --- | --- | --- |
| `Clan` | Lightweight Community base record and compatibility anchor. | `CommunityDomain` for paid institutional identity and lifecycle. |
| `ClanMembership` | Basic membership in a community. | Node membership and domain role/policy assignment. |
| `ClanInvite` / `ClanJoinRequest` | Existing invite and join-request flow. | Domain-aware invites tied to node, role, policy, and approval route. |
| `CommunityDomainAffiliation` | Parent/affiliate acknowledgement spine. | Link affiliation to Community Domains once the durable domain object exists. |
| `CommunityMemberVerification` | Member witness/currentness evidence. | Domain evidence records and role/node-specific evidence. |
| `FeatureEntitlement` | Paid feature units/credits. | Community Domain subscription lifecycle and module entitlement status. |
| `/payment-instructions/community-package` | Current package payment rail. | New Community Domain package quote/payment flow that does not activate until confirmed. |
| Marketplace shop/spotlight/vault routes | Existing commerce and visibility tools. | Domain-aware visibility tables and trust relay paths. |
| TrustSlip / Trust Passport | Portable member evidence. | Domain-scoped evidence summaries and currentness signals. |

Devil's advocate:

- If we only rename existing `community-package` bundles, schools and churches
  still will not have structure, governance, node roles, appeal paths, or
  domain identity.
- If we create a new institutional engine without reusing the existing spines,
  we risk duplicating identity, membership, payment, and evidence logic.
- The clean middle path is a Community Domain layer that references the current
  community/member/payment/trust spines while adding the institutional
  lifecycle and hierarchy they currently lack.

## 7. Shared GSN Services Become Domain-Aware

The Community Domain should not replace existing services. It should add a
community-aware operating layer around them.

| Service | Domain-aware behavior |
| --- | --- |
| Shops | Owner chooses which Community Domains can see the shop or selected blocks. |
| Spotlight | Owner can publish inside own domain or pay/relay into permitted domains. |
| Vault | Owner shares private blocks with approved members, nodes, or invited viewers. |
| Demand Box | Demand can be scoped to a domain, node, or permitted cross-domain audience. |
| Verification | Evidence ties back to domain membership, role, and node context. |
| Trust Passport | One member identity carries many community/domain contexts. |
| TrustSlip | Portable evidence can include relevant domain membership evidence. |
| Notifications | Audiences can be domain-wide, node-specific, role-specific, or invite-specific. |
| Analytics | Shows membership, module activity, trust relay, shop/spotlight reach, and evidence currentness. |

## 7.1 Society-Fit Template Framework

Community Domains should work globally by changing presets, not by changing
the engine.

| Society type | Typical nodes | Core modules | Marketplace role |
| --- | --- | --- | --- |
| School | campus, class, department, PTA, alumni, staff unit | members, structure, events, vault, verification, analytics | optional: uniforms, books, parent vendors, alumni support |
| Church / religious body | branch, ministry, choir, youth, welfare, usher, cell group | members, structure, events, trust centre, welfare evidence, announcements | optional: member businesses, welfare fundraising, approved vendors |
| Union / professional body | branch, chapter, committee, profession grade, welfare unit | members, roles, credential evidence, verification, announcements, analytics | useful: job referrals, member services, trusted vendors |
| Market / cooperative | section, line, product group, stall cluster, ROSCA group | shops, spotlight, demand, verification, ROSCA, trust relay, analytics | central |
| Family / town union / diaspora | family branch, village unit, project committee, welfare group | members, vault, events, contributions evidence, trust centre | optional: member businesses and project sourcing |
| Hospital / health body | department, unit, role grade, shift group | roles, verification, controlled documents, events, analytics | limited/controlled |
| NGO / project network | project, field office, beneficiary group, partner unit | evidence, vault, reports, member roles, analytics | optional: suppliers and livelihood programs |
| Generic association | branch, committee, member class | members, structure, roles, events, verification | configurable |

Each template should define:

- default node labels;
- default role labels;
- default enabled modules;
- default approval policy;
- default public/private evidence rules;
- default dashboard first action.

Important:

- templates are presets, not separate schemas;
- admins can rename node labels without changing the backend;
- marketplace should be optional for non-market domains, but it should never be
  removed from the platform because many institutional domains will eventually
  use it for trusted vendors, welfare, fundraising, jobs, services, or member
  businesses.

## 7.2 Minimum Full Package

For a Community Domain to feel like a complete product, not a renamed community
screen, the first sellable package should include:

1. Domain identity
   - domain name, public profile, owner, status, renewal state.

2. Structure
   - branches/departments/classes/committees as generic nodes.

3. Membership
   - request, invite, approve, reject, assign to node, assign role.

4. Governance
   - roles, policy presets, action reviews, appeals.

5. Modules
   - shop, spotlight, vault, demand, events, verification, notifications,
     analytics as enabled/disabled settings.

6. Evidence
   - domain evidence records, member witness currentness, role appointments,
     node membership evidence, public-safe summaries.

7. Trust mobility
   - Trust Passport and TrustSlip can show domain-scoped evidence without
     merging private institutional records into a universal score.

8. Renewal and limits
   - active, expired, suspended, closed states; paid features restrict when the
     domain is expired but history remains readable to authorized users.

## 9. Acceptance Criteria

The Community Domain package is not real until all of this is true:

- an owner can start `Purchase Community Domain`;
- domain name availability is checked;
- a payment instruction can be generated;
- domain is not active until payment/admin confirmation;
- activation provisions identity, owner role, settings, and root node;
- admin can create branches/departments/classes/committees through generic
  nodes;
- members can request entry;
- admins can approve, reject, assign roles, and assign nodes;
- one GSN member can belong to many communities/domains with one identity;
- shops, spotlight, vault, verification, and notifications can be scoped to the
  domain;
- a relay/repost path records source, bridge member, and destination domain;
- public viewers see safe domain evidence without private records;
- expired/suspended domains keep history but lose paid operating rights until
  renewed.

## 10. First Safe Code Move

The first implementation code slice should be backend-only and test-led:

1. Add the `CommunityDomain`, `CommunityNode`, and template-key fields behind
   migration.
2. Add domain availability and draft creation routes.
3. Add tests proving:
   - domain names are unique;
   - draft is not active;
   - only the owner/admin can continue provisioning;
   - existing lightweight community creation still works.

Then add policy/action-review and node membership in the second backend slice.
Do not put all governance tables into the first migration if that makes review
too large.

Do not start with the dashboard UI. The UI would look impressive but could
hide the absence of a real billing/provisioning object.

## 11. Open Product Decisions

These must be decided before pricing/UI copy hardens:

- Package tiers: one starter institutional package or multiple tiers?
- Renewal period: monthly, yearly, or both?
- Can an existing lightweight community be upgraded in place?
- Should the public URL be `/domains/:name` or `/community-domains/:name`?
- Which modules are included by default?
- Does domain verification require documents, admin approval, member witness,
  parent-domain affiliation, or some combination?
- What happens to member access when a domain expires?

Until those are answered, code should keep defaults conservative and avoid
claims that imply legal certification, payment movement, loan approval, or
guaranteed trust.
