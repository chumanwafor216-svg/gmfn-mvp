# GSN Book Completion Protocol Draft

Date: 2026-07-04

Status: Draft for owner review

## Purpose

This draft records the protocol for bringing the live GSN product as close as
possible to full alignment with the attached institutional handbook.

The owner's position is that GSN already contains roughly 95 percent of the
book's concepts. The working truth is more precise:

- GSN is already very close in philosophy, vocabulary, and domain structure.
- GSN is not yet equally complete in user-facing clarity, operational proof,
  institutional maturity, and network scale.
- The remaining work is mostly alignment, evidence standardization, route
  discipline, and proof of operation rather than invention of new product ideas.

The book's central chain is:

Community -> Behaviour -> Evidence -> Trust Event -> Trust Passport ->
TrustSlip -> Decision -> Opportunity -> Prosperity

Every major GSN screen, route, action, and data object should either support
that chain directly or clearly belong to a secondary support role.

## Completion Standard

GSN reaches full book alignment when:

1. Every major book concept has a named product equivalent.
2. Every product equivalent has a route, component, data model, or documented
   roadmap state.
3. Every major screen makes the evidence-to-decision purpose clear.
4. Private information stays private while public evidence remains useful.
5. No page claims a capability is live unless it is actually live.
6. Advanced institutional claims are marked as pilot, prepared, roadmap, or
   institutional future where needed.
7. A new user can understand GSN's purpose from the app without reading the
   book.

## Truth Rule

GSN must not confuse vision with current reality.

Use these maturity labels in docs and selectively in UI:

| Status | Meaning |
| --- | --- |
| Live | Built, usable, and connected to current product behavior. |
| Pilot | Built or partly built, but still being tested with limited users. |
| Prepared | Designed in product structure, but not fully activated. |
| Roadmap | Approved direction, not yet implemented. |
| Institutional Future | Long-term ambition requiring adoption, partners, or scale. |

This lets GSN carry the full institutional vision without overclaiming.

## Book-To-Product Mapping

| Book Concept | GSN Product Equivalent | Completion Need |
| --- | --- | --- |
| Community | Community Home / Community Domain | Clear identity, purpose, members, records, and owner actions. |
| Behaviour | Member, shop, and community actions | Actions should become recordable evidence, not just UI activity. |
| Evidence | Public Shelf, Private Vault, Community Record | Evidence must be categorized, visibility-aware, and decision-useful. |
| Trust Event | Standard evidence event | Events need source, date, actor, visibility, provenance, and meaning. |
| Trust Passport | My GSN Identity | Must feel like portable trust identity, not a normal profile. |
| TrustSlip | Shareable evidence snapshot | Must explain what is shared, why, and what remains private. |
| Community Memory | Community Record | Must preserve useful witnessed history without becoming an activity dump. |
| Behavioural Capital | Demonstrated trust value | Should be framed carefully, not prematurely financialized. |
| Community Intelligence | Privacy-safe aggregated insight | Should remain pilot/roadmap until real aggregation exists. |
| Opportunity | Marketplace, borrowing, shop access, verification confidence | Opportunities should be tied to evidence, not generic listings. |
| Prosperity | Safer trade, access, confidence, growth | Should be shown as possible outcomes, not guaranteed promises. |

## Screen Alignment Questions

Each screen should answer one primary book-aligned question.

| Screen | Primary Question |
| --- | --- |
| Dashboard | What matters now in my trust life? |
| My GSN Identity | What has been preserved about my demonstrated trust? |
| GSN & I / Capabilities | What can GSN help me decide or prove? |
| Community Home | What trust memory does this community preserve? |
| Community Record | What has this community witnessed or confirmed? |
| Marketplace | Which opportunities are supported by evidence? |
| Public Shop | What evidence supports this seller before trade? |
| Private Vault | What sensitive evidence is protected but available when needed? |
| TrustSlip | What evidence can I safely share for this decision? |
| Confirmation Policy | Who may help confirm, and what privacy is protected? |

Any screen that cannot answer its question should be simplified, collapsed, or
reframed.

## UI Completion Rules

The product should not become heavier to mirror the book. It should become
cleaner.

1. One primary action per screen.
2. Long doctrine belongs in expandable guides, not exposed page bodies.
3. Each card should make a decision easier.
4. Each badge should represent evidence, status, or permission.
5. Each public page should separate visible proof from protected private data.
6. Avoid explaining the whole system on every page.
7. Use calm, precise, institutional language.
8. Collapse secondary material behind rows, drawers, or accordions.
9. Do not expose all capabilities at once.
10. Make the next best action obvious.

## Evidence Completion Protocol

Every evidence item should eventually have:

- Evidence title
- Evidence type
- Source
- Community or member context
- Date or period
- Visibility level
- Decision relevance
- Verification state
- Related Trust Passport section
- Related TrustSlip eligibility
- Privacy warning where needed

This is the bridge from philosophy to infrastructure.

## Trust Event Protocol

A Trust Event is a preserved moment of behaviour that can support future
decisions.

Each Trust Event should answer:

1. What happened?
2. Who or what confirmed it?
3. Which community context does it belong to?
4. Is it public, private, or shareable by permission?
5. What decision could it support later?
6. Has it expired, changed, or been challenged?

## Trust Passport Completion Protocol

The Trust Passport should become the member's portable trust identity.

It should include:

- GSN ID
- Community memberships
- Confirmation readiness
- Public evidence
- Private evidence summaries
- Shop identity link
- TrustSlip sharing controls
- Capability readiness
- Warnings or missing evidence
- Recent Trust Events
- Decision-specific proof paths

The Passport should not feel like a profile. It should feel like a living
evidence instrument.

## TrustSlip Completion Protocol

A TrustSlip should never be only a share link.

It should say:

- What decision the TrustSlip supports
- What evidence is included
- What evidence is excluded
- What private information remains hidden
- Who issued it
- When it was created
- Whether it is still valid
- What the receiver should and should not conclude

## Marketplace Completion Protocol

The marketplace must not behave like a generic listing board.

Every listing or shop should show:

- Seller GSN ID
- Community context
- Public evidence available
- Verification state
- TrustSlip or evidence action
- What remains unverified
- Safe next action

The marketplace should train the user to ask: what evidence supports this trade?

## Community Domain Completion Protocol

A Community Domain should represent an operating trust environment, not just a
group page.

It should include:

- Community identity
- Community purpose
- Members or member count
- Owner/admin actions
- Community Record
- Evidence standards
- Public/private boundary
- Marketplace or shop links
- Confirmation policy
- Domain maturity status

## Behavioural Capital Guardrail

The book's Behavioural Capital concept is powerful but risky if framed too early
as money, credit, investment value, or guaranteed opportunity.

Safer product framing:

Demonstrated trust value built from repeated evidence.

Use this concept behind trust readiness, seller confidence, community
reliability, borrowing readiness, verification strength, and opportunity
eligibility. Do not present it as a financial asset without legal, operational,
and product backing.

## Completion Phases

### Phase 1: Alignment Map

Create a detailed `docs/GSN_BOOK_ALIGNMENT_MAP.md` listing:

- Book concept
- Existing product equivalent
- Route or file
- Status: Live, Pilot, Prepared, Roadmap, Institutional Future
- Gap
- Next action

### Phase 2: Screen Purpose Audit

Audit authenticated screens and assign each one a book question. Compress
screens that expose too much. Sharpen screens that hide their purpose.

### Phase 3: Evidence Standardization

Define shared Trust Event and evidence display contracts so evidence stops
appearing as disconnected page content.

### Phase 4: Trust Passport Upgrade

Make My GSN Identity feel like the true Trust Passport command center:
identity, evidence, privacy, sharing, decision readiness, and missing proof.

### Phase 5: TrustSlip Upgrade

Make TrustSlip flows decision-specific: seller verification, membership
confirmation, borrowing readiness, community record proof, shop credibility,
payout decision, or protected release decision.

### Phase 6: Marketplace Evidence Upgrade

Make public shops and marketplace cards evidence-first, with clear boundaries
between confirmed, private, and still-unverified claims.

### Phase 7: Community Memory Upgrade

Make Community Record the visible form of Community Memory. It should be
curated proof of what the community has witnessed, not a generic activity log.

### Phase 8: Institutional Maturity Labels

Add maturity labels in docs first, then selectively in UI. This keeps ambition
honest.

### Phase 9: Pilot Proof Loop

Track real usage:

- Confirmations requested
- TrustSlips created
- Shops verified
- Evidence added
- Communities activated
- Marketplace decisions supported

This is how the book becomes proven operation instead of only doctrine.

## Definition Of Full Book Alignment

GSN is fully aligned with the book when every major concept in the book exists
in the product as a live feature, pilot feature, prepared structure, or clearly
marked roadmap layer, and every live screen helps users move from community
behaviour to preserved evidence to better decisions without unnecessary exposure
of private information.

## Devil's Advocate

The risk is not that GSN lacks the book. The risk is that GSN already has so
much of the book that the product may become dense, reverent, or too
explanatory.

The final work is discipline:

- Fewer exposed blocks
- Clearer evidence
- Stronger decision framing
- Honest maturity labels
- Better route-by-route consistency
- Less explaining GSN
- More using GSN to decide safely
